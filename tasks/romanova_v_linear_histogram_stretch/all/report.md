## 1. Контекст
Гибридная версия (MPI + OpenMP) требуется для совмещения двух уровней параллелизма:
- **Межпроцессный (MPI):** позволяет распределить данные между несколькими процессами, что даёт возможность обрабатывать изображения, не помещающиеся в память одного узла, и использовать вычислительные ресурсы кластера.
- **Внутрипроцессный (OpenMP):** обеспечивает эффективную утилизацию всех ядер каждого узла за счёт легковесных потоков внутри одного процесса.

Такой подход особенно полезен для обработки сверхбольших изображений (терапиксели), где одного узла недостаточно, либо для гетерогенных систем, где узлы имеют разное количество ядер. Кроме того, гибридная модель позволяет снизить нагрузку на MPI-коммуникации: обмен данными происходит только между процессами, а потоки внутри процесса обмениваются данными через общую память, что значительно быстрее.

## 2. Постановка задачи
Задача — повысить контрастность входного изображения за счёт выполнения линейного растяжения гистограммы. 
В качестве базовой линии (baseline) используется результат последовательной версии из `seq/report.md` (время выполнения:  0.81182274 с).

Детали постановки задачи можно найти в `seq/report.md` в разделе `2. Постановка задачи`.

## 3. Базовый алгоритм
Последовательная версия за один проход находит минимальное и максимальное значения пикселей во входном массиве, после чего за второй проход преобразует каждый пиксель по формуле линейного растяжения: $out[i] = \frac{(in[i] - min_v)*255}{max_v-min_v}$. Если `min == max` (монохромное изображение), то возвращается копия входного массива без преобразования.

## 4. Межпроцессная схема

### Роли rank-ов

- **Rank 0 (главный процесс):** выполняет инициализацию — проверяет входные данные, вычисляет параметры распределения (`counts` и `displs`), рассылает их всем процессам. А также участвует в распределении исходных данных и сборе результата.
- **Rank 1...N-1 (вспомогательные процессы):** получают параметры от rank 0, принимают свою часть данных, выполняют локальную обработку, участвуют в глобальных редукциях и возвращают обработанные данные.

### Этап 1: Валидация

На этапе валидации rank 0 проверяет, что входной массив не пуст. Затем логическое значение статуса рассылается всем процессам через коллективную операцию `MPI_Bcast`, чтобы все процессы получили корректное значение состояния.

### Этап 2: Препроцессинг

1. **Синхронизация размера:** rank 0 передаёт размер входного массива всем процессам через `MPI_Bcast`. На основе этой информации каждый процесс вычисляет размер своего локального блока  в соответствии с формулой `local_size_ = delta + (std::cmp_less(rank, extra) ? 1 : 0);`.

2. **Вычисление и рассылка вспомогательных данных:** на rank 0 рассчитываются `vector_counts` (размер части локального блока для каждого процесса) и `vector_displs` (смещения для последующей рассылки и сборки данных). Оба массива рассылаются всем процессам через `MPI_Bcast`.

3. **Распределение данных:** при помощи `MPI_Scatterv` выполняется отправка данных от rank 0 всем процессам. Каждый процесс получает свою непрерывную часть исходного массива в локальный буфер и далее взаимодействует только с ним.

### Этап 3: Выполнение

1. **Поиск min/max:** каждый процесс независимо находит локальные минимум и максимум в своей части данных. Далее через `MPI_Allreduce` найденные локальные значения редуцируются и рассылаются по всем процессам, в результате чего каждый процесс получает глобальное значение максимума.

3. **Преобразование:** каждый процесс независимо преобразует свою часть данных, используя глобальные min/max.

### Этап 4: Постпроцессинг

При помощи операции `MPI_Allgatherv` данные, рассчитанные в локальных буферах, собираются вместе и рассылаются по всем процессам.

### Синхронизация

Коллективные операции MPI (`Bcast`, `Scatterv`, `Allreduce`, `Allgatherv`) сами обеспечивают неявную синхронизацию между процессами. Явные барьеры (`MPI_Barrier`) не используются, так как в них нет необходимости.

## 5. Внутрипроцессная схема
- **Используемая технология:** внутри каждого MPI-процесса используется **OpenMP** для распараллеливания работы на уровне потоков. Это позволяет эффективно утилизировать все ядра каждого узла.

- **Разбиение работы между потоками:**
  - Данные, полученные процессом (локальная часть `local_data_`), разбиваются на непересекающиеся диапазоны с помощью `#pragma omp parallel for` с распределением `schedule(static)` по умолчанию.

- **Объединение локальных результатов внутри процесса:**
  - Для поиска min/max используется конструкция `reduction(min:loc_min_v, max:loc_max_v)`, которая создаёт локальные копии переменных для каждого потока. После завершения цикла OpenMP автоматически объединяет эти локальные значения с помощью операций `min` и `max`.
  - При преобразовании объединение не требуется, так как каждый поток независимо записывает результат в свою часть выходного буфера `local_out_`.

- **Синхронизация внутри процесса:**
  - Неявные барьеры в конце каждой директивы `parallel for` обеспечивают синхронизацию потоков.


## 6. Детали реализации

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.

### MPI-вызовы и их расположение

| MPI-вызов | Местоположение | Назначение |
|-----------|----------------|------------|
| `MPI_Comm_rank` / `MPI_Comm_size` | `ValidationImpl`, `PreProcessingImpl` | Определение ранга и числа процессов |
| `MPI_Bcast` | `ValidationImpl`, `PreProcessingImpl` | Рассылка статуса, размера, counts/displs |
| `MPI_Scatterv` | `PreProcessingImpl` | Распределение данных от rank 0 |
| `MPI_Allreduce` | `RunImpl` | Глобальный поиск min/max |
| `MPI_Allgatherv` | `PostProcessingImpl` | Сбор результатов от всех процессов |

**Синхронизация:** коллективные операции MPI сами обеспечивают синхронизацию. Явный `MPI_Barrier` не используется.

### Потенциальные узкие места

1. **Передача больших объёмов данных:** `MPI_Scatterv` и `MPI_Allgatherv` передают весь массив через сеть, что создаёт значительную коммуникационную нагрузку.

2. **Два последовательных `MPI_Allreduce`:** две отдельные редукции (min и max) удваивают коммуникационные затраты.

3. **Сбор данных на каждом процессе:** `MPI_Allgatherv` требует, чтобы каждый процесс получил полный результат, что избыточно при большом количестве процессов.

4. **Memory-bound локальных вычислений:** внутри каждого процесса задача ограничена пропускной способностью памяти.


## 7. Проверка корректности
Корректность проверялась сравнением с последовательной и параллельными версиями на стандартном наборе из 6 функциональных тестов. Проверка на согласованность показала,
что после финального `MPI_Allreduce` все процессы возвращают идентичный массив, для которого выполняется одно из двух условий:
- Выходное изображение монохромное (`min == max`) — тогда все пиксели остаются

- Выходное изображение имеет полный динамический диапазон: минимальное значение
  равно 0, максимальное значение равно 255.

### Анализ результатов и масштабируемости

1. **Эффективность гибридной схемы:** Использование MPI+OpenMP позволило сократить время обработки изображения с **0.81 с** до **0.30 с**. Максимальное достигнутое ускорение составило **2.67×** при использовании 32 работников (4 процесса × 8 потоков).

2. **Низкая масштабируемость:** При увеличении общего числа работников с 4 до 32 ускорение выросло лишь с 2.36× до 2.67×. Это свидетельствует о том, что гибридная версия сильно ограничена коммуникационными накладными расходами.

3. **Влияние числа процессов:** Переход с конфигурации 2×2 (4 работника) на 4×2 (8 работников) дал незначительный прирост ускорения (2.36 → 2.42). Добавление новых MPI-процессов без увеличения нагрузки на каждый процесс неэффективно.

4. **Влияние числа потоков на процесс:** Увеличение числа потоков с 2 до 8 на 4 процессах (с 8 до 32 работников) привело к скромному росту ускорения с 2.42 до 2.67. Это говорит о том, что задача внутри процесса также ограничена пропускной способностью памяти.

5. **Основное узкое место:** Коммуникационные затраты на `MPI_Scatterv`, `MPI_Allreduce` и `MPI_Allgatherv` нивелируют выигрыш от параллельных вычислений. Передача больших объёмов данных между процессами становится доминирующим фактором.

## 10. Выводы

Гибридная схема MPI+OpenMP показала низкую эффективность для данной задачи. Коммуникационные затраты на передачу данных между процессами и глобальные редукции превышают выигрыш от параллельных вычислений, что привело к ускорению всего **2.67×** при использовании 32 работников. Увеличение числа потоков внутри процесса даёт лишь незначительное улучшение из-за того, что задача ограничена пропускной способностью памяти.

Применение гибридной схемы оправдано только в распределённых системах (кластерах) при работе с экстремально большими объёмами данных, которые не помещаются в память одного узла. Для задач, умещающихся в ОЗУ одной машины, MPI+OpenMP неэффективна из-за оверхеда на межпроцессное взаимодействие. В таких случаях оптимальным решением является использование чистой thread-версии (OpenMP/TBB/STL) на одном узле, где отсутствуют коммуникационные накладные расходы.
