## 1. Контекст
Из последовательной версии (SEQ) в OpenMP переносится вся вычислительная логика линейного растяжения гистограммы: поиск минимального и максимального значений пикселей, а затем линейное преобразование каждого пикселя в диапазон [0, 255].

Перенос в среду OpenMP необходим для разделения исходного массива на независимые части и их параллельной обработки на многоядерной архитектуре. Оба этапа алгоритма обладают естественным параллелизмом:

- **Поиск min/max** — редукционная операция, где каждый элемент массива обрабатывается независимо, а результаты (локальные минимумы и максимумы) затем объединяются. OpenMP предоставляет конструкцию `reduction`, которая идеально подходит для таких задач.

- **Преобразование пикселей** — операция, где каждый выходной пиксель зависит только от соответствующего входного пикселя и глобальных min/max. Отсутствие зависимостей между итерациями позволяет полностью распараллелить цикл преобразования.

## 2. Постановка задачи
Задача — повысить контрастность входного изображения за счёт выполнения линейного растяжения гистограммы. 
В качестве базовой линии (baseline) используется результат последовательной версии из `seq/report.md` (время выполнения:  0.81182274 с).

Детали постановки задачи можно найти в `seq/report.md` в разделе `2. Постановка задачи`.

## 3. Базовый алгоритм
Последовательная версия за один проход находит минимальное и максимальное значения пикселей во входном массиве, после чего за второй проход преобразует каждый пиксель по формуле линейного растяжения: $out[i] = \frac{(in[i] - min_v)*255}{max_v-min_v}$. Если `min == max` (монохромное изображение), то возвращается копия входного массива без преобразования.

## 4. Схема распараллеливания
Для реализации OMP-версии выбрана следующая схема распараллеливания:

- **Декомпозиция данных:** массив пикселей делится на равные части автоматически директивой `#pragma omp parallel for` с распределением `schedule(static)` по умолчанию. Каждый поток получает непрерывный диапазон итераций. Все итерации цикла имеют одинаковую вычислительную сложность, следовательно динамическое распределение не требуется.

- **Параллельный поиск min/max:** используется директива `#pragma omp parallel for` с редукцией. Каждый поток обрабатывает свой чанк, вычисляя локальные минимум и максимум. После завершения цикла локальные значения автоматически объединяются в глобальные `min_v` и `max_v` с помощью операций `min` и `max`.

- **Параллельное преобразование:** после получения глобальных `min_v` и `max_v` запускается второй параллельный цикл с директивой `#pragma omp parallel for`. Каждый поток преобразует пиксели своего куска данных по формуле линейного растяжения: `out[i] = (in[i] - min_v) * 255 / (max_v - min_v)`.

- **Обработка монохромного случая:** если `min_v == max_v` (все пиксели одинаковы), преобразование заменяется на параллельное копирование данных из входного массива в выходной.

- **Переменные:**
  - `shared`: входной массив (`in`), выходной массив (`out`), размер массива (`size`), глобальные `min_v`, `max_v`, коэффициент `ratio`.
  - `private`: индекс цикла `i` (автоматически при `parallel for`), временная переменная `pix`.

- **Редукция**: используется `reduction(min : min_v) reduction(max : max_v)` в первой фазе — каждый поток работает с локальной копией `min_v`, `max_v`, после цикла значения объединяются.

- **Синхронизация:** неявные барьеры в конце каждой директивы `parallel for` обеспечивают синхронизацию между фазами. Атомарные операции и критические секции не требуются, так как:
  - При поиске min/max редукция полностью решает проблему синхронизации.
  - При преобразовании и копировании потоки работают с непересекающимися областями памяти (каждый пиксель записывается только одним потоком).


## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

- **Изменения относительно SEQ:**
    1. Поиск min/max переписан с использованием `#pragma omp parallel for` с редукцией вместо `std::ranges::minmax_element`.
    2. Цикл преобразования пикселей выполняется под управлением `#pragma omp parallel for` для параллельной обработки каждого чанка.
    3. Цикл копирования для монохромного случая также распараллелен с помощью `#pragma omp parallel for`.

- **Риски гонок** устранены за счет:
    1. Использования `reduction(min:min_v, max:max_v)` при поиске min/max гарантирует, что каждый поток работает с локальной копией, затем значения автоматически объединяются.
    2. Распределения `schedule(static)`, которое гарантирует, что каждый поток пишет в свой уникальный диапазон выходного массива.

## 6. Проверка корректности

Результат работы OMP-версии считается корректным при выполнении одного из двух условий:
- Выходное изображение монохромное (`min == max`) — тогда все пиксели остаются
  неизменными.
- Выходное изображение имеет полный динамический диапазон: минимальное значение
  равно 0, максимальное значение равно 255.

Помимо этого производилось сравнение с работой SEQ версии. Ошибок в логике работы алгоритма выявлено не было.


Набор функциональных тестов:
1. **Изображение 1×1 (один пиксель):** проверка граничного случая
   минимального размера и ветки `min_v == max_v`.

2. **Изображение 5×5:** проверка
   правильности растяжения на малом размере.

3. **Изображение 13×13:** проверка работы алгоритма на среднем размере входных данных.

4. **Изображение 70×70:** проверка устойчивости при большем количестве
   пикселей.

5. **Изображение 200×200:** проверка корректности
   на значительном объёме данных.

6. **Реальное монохромное изображение:** — проверка ветки `min_v == max_v` на реальных данных,
   загружаемых из файла.

### Анализ результатов и масштабируемости

На основе полученных данных можно сделать следующие выводы:

1. **Наличие ускорения:** Переход от последовательной версии к параллельной на 2 потоках даёт ускорение $S = 2.52$,
   что превышает идеальное значение 2.0. Это свидетельствует о суперлинейном ускорении, которое возможно при
   эффективном использовании кэш-памяти: при последовательной обработке данные могут вытесняться из кэша, а при
   параллельной — каждый поток обрабатывает свой диапазон, который лучше попадает в кэш L1/L2.

2. **Эффективность:** При использовании 2 потоков эффективность составляет $1.26$, что подтверждает
   эффект суперлинейного ускорения. При 4 потоках эффективность снижается до $0.72$, что всё ещё является хорошим
   показателем. Снижение относительно идеальной единицы обусловлено накладными расходами на создание параллельных
   регионов и синхронизацию при выполнении `reduction`.

3. **Предел масштабируемости (Закон Амдала):** При переходе с 4 на 8 потоков прирост ускорения минимален
   ($2.88 \rightarrow 3.09$), а эффективность падает до $0.39$. Это свидетельствует о том, что для изображения
   данного размера пропускная способность памяти становится основным ограничивающим фактором.

4. **Узкие места:** Алгоритм линейной растяжки гистограммы имеет низкую вычислительную плотность
   (мало операций на одно чтение из памяти). Основным ограничением выступает пропускная способность памяти при
   одновременном доступе нескольких ядер к массиву данных, что характерно для задач обработки изображений.

5. **Влияние архитектуры:** Суперлинейное ускорение на 2 потоках и последующее насыщение на 4-8 потоках может
   быть связано с особенностями кэш-иерархии процессора и распределением данных между ядрами.

**Заключение:** Для текущей задачи оптимальным является использование 2-4 потоков. На 2 потоках достигается
суперлинейное ускорение за счёт кэш-эффектов, а на 4 потоках обеспечивается наилучший баланс между ускорением
($2.88\times$) и эффективностью использования ресурсов ($72\%$). Дальнейшее увеличение числа потоков нецелесообразно
из-за насыщения пропускной способности памяти.

OMP-версия даёт выигрыш при параллельной обработке независимых чанков данных

использование 2-4 потоков в зависимости от архитектуры целевой системы.
