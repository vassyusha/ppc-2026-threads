## 1. Введение

**Линейное растяжение гистограммы** — это базовый метод улучшения изображений, который преобразует пиксели так, чтобы выходные значения заполняли весь диапазон [0, 255]. Алгоритм состоит из двух этапов: поиск глобальных минимума и максимума, затем линейное преобразование каждого пикселя.

Данная задача хорошо подходит для сравнения разных моделей параллелизма по нескольким причинам:

- **Естественный параллелизм:** поиск min/max является редукционной операцией, а преобразование пикселей — независимыми вычислениями, что позволяет эффективно распараллелить оба этапа.

- **Низкая вычислительная плотность:** на каждый пиксель приходится всего несколько операций, что делает задачу memory-bound. Это позволяет оценить, насколько разные технологии справляются с ограничениями пропускной способности памяти.

- **Множество вариантов реализации:** задача может быть решена с использованием OpenMP, TBB, STL Threads и гибридной MPI+OpenMP, что даёт возможность сравнить их на одном алгоритме.

- **Простота верификации:** корректность результата легко проверить — выходное изображение должно иметь диапазон [0, 255] или быть монохромным.

Таким образом, линейное растяжение гистограммы позволяет на реальном примере оценить производительность, удобство реализации и масштабируемость различных технологий параллельного программирования.

## 2. Единая постановка задачи
**Входные данные:** одномерный массив `uint8_t` пикселей изображения (значения в диапазоне 0–255).

**Выходные данные:** массив того же размера после линейного растяжения гистограммы. Минимальное значение выходного массива — 0, максимальное — 255.

Растяжение происходит поэлементно в соответствии с формулой $out[i] = \frac{(in[i] - min_v)*255}{max_v-min_v}$.

**Ограничения:** входной массив не пуст.

Результат работы программы считается корректным, если выполняется одного из двух условий:
- Выходное изображение монохромное (`min == max`) — тогда все пиксели остаются
  неизменными.
- Выходное изображение имеет полный динамический диапазон: минимальное значение
  равно 0, максимальное значение равно 255.

### Замеры производительности
Для оценки производительности запускались тесты на векторе из **400 000 000** элементов типа `uint_8t`.

**Генерация данных** происходит по формуле `in[i][j] = low + ((j * 13 + i * 109) % range)`, где `low = 75`, `range = 50`. 
Т.е. все значения исходного массива данных лежат в диапазоне [75, 124].

Замеры времени производились следующим образом:
1. Производится 5 запусков задачи.
2. Из 5 запусков откидываются лучшее и худшее время.
3. Среднее арифметическое из 3 оставшихся значений записывается в таблицу.

Параметры ускорения и эффективности высчитываются в соответствии со следующими формулами:
- $speedup = T_{seq} / T_th$ — ускорение для установленного числа потоков.
- $efficiency = speedup / threads$ — эффективность использования вычислительных ресурсов.

4. **Изображение 70×70:** проверка устойчивости при большем количестве

5. **Изображение 200×200:** проверка корректности

6. **Реальное монохромное изображение:** — проверка ветки `min_v == max_v` на реальных данных,

## 6. Интерпретация различий

### 6.1. SEQ

SEQ задаёт точку отсчёта (`T_seq = 0.81` с) и демонстрирует ключевые характеристики задачи: низкую арифметическую интенсивность, линейный доступ к памяти и независимость пикселей, что делает её потенциально хорошо распараллеливаемой.

### 6.2. OMP

**Сильные стороны:** Минимальные изменения кода (добавление директив `parallel for` и `reduction`), удобный механизм редукции для поиска min/max.

**Слабые стороны:** Падение эффективности до 39% на 8 потоках объясняется использованием Hyper-Threading (HT). Процессор имеет 4 физических ядра, каждое из которых выполняет 2 логических потока. При переходе с 4 на 8 потоков начинают использоваться виртуальные (HT) потоки, которые разделяют ресурсы физического ядра (кэш L1/L2, исполнительные блоки), что даёт лишь ограниченный прирост производительности.

### 6.3. TBB

**Роль grain size:** Автоматический выбор размера зерна избавляет от ручной настройки и позволяет TBB достичь наилучшего ускорения 5.45×. Платой являются дополнительные вызовы планировщика, но в данной задаче они окупаются.

**Роль runtime:** TBB показал лучший результат (5.45× на 8 потоках), эффективно балансируя нагрузку между 4 физическими ядрами и их HT-потоками. Механизм _work stealing_ позволяет сглаживать дисбаланс, возникающий при использовании HT.

### 6.4. STL

**Цена ручного управления:** Увеличенный объём кода, необходимость ручного вычисления диапазонов, явный `join()`, потенциальные риски гонок, отсутствие динамической балансировки.

**Выгода:** Минимальные накладные расходы, полный контроль над разбиением данных, предсказуемое поведение.

**Почему STL показал хороший результат (4.76× на 8 потоках):** Статическое разбиение данных на 8 равных частей хорошо работает при использовании HT, так как каждый логический поток получает свой непрерывный блок данных, что минимизирует конфликты в кэше.

### 6.5. ALL

**Цена коммуникации:**
- `MPI_Scatterv`/`MPI_Allgatherv` требуют передачи всего массива данных через коммуникационную сеть.
- `MPI_Allreduce` (два вызова) заставляет все процессы ждать самого медленного.
- Даже на одном узле межпроцессное взаимодействие создаёт избыточные копирования данных.

**Дополнительный фактор:** Помимо HT-ограничений, добавляются коммуникационные накладные расходы MPI, что делает гибридную версию наименее эффективной на одном узле.

**Когда ALL оправдан:**
- Данные не помещаются в память одного узла (терапиксельные изображения).
- Реальный многоузловой кластер с высокоскоростной сетью (InfiniBand).
- Задачи с высокой вычислительной сложностью на пиксель, где коммуникации незаметны на фоне вычислений.

## 8. Заключение

### Какая версия лучше и когда

На исследованной системе **наилучшую производительность показала версия на TBB** — ускорение 5.45× при использовании 8 потоков.

Выбор подходящей технологии зависит от поставленных задач:

- **OpenMP** — оптимальный выбор для быстрого прототипирования. Требует минимальных изменений в коде (всего несколько директив) и даёт ускорение до 3.09×.

- **TBB** — лучший вариант, когда требуется выжать максимум из одного узла. Библиотека самостоятельно балансирует нагрузку и хорошо адаптируется к Hyper-Threading, обеспечивая ускорение 5.45×.

- **STL Threads** — подходит для ситуаций, где важен полный контроль над потоками. При статическом разбиении данных даёт ускорение 4.76× с минимальными накладными расходами.

- **MPI+OpenMP** — имеет смысл только в распределённых системах при обработке данных, которые не помещаются в память одного узла. На одиночной машине коммуникационные издержки делают эту версию неэффективной.

- **SEQ** — служит эталоном для проверки корректности и измерения базовой производительности.

### Что ограничивает достоверность сравнения

Полученные цифры привязаны к конкретному оборудованию (процессор i5-1135G7). На других системах результаты могут отличаться.

Эксперименты проводились на фиксированном размере данных (400 млн пикселей). При меньших объёмах накладные расходы на параллелизацию могут свести на нет выигрыш от многопоточности.

Задача является memory-bound, то есть упирается в пропускную способность памяти. Для вычислительно-ёмких алгоритмов соотношение эффективности технологий может быть иным.

OpenMP и TBB не подвергались тонкой настройке — использовались параметры по умолчанию. Более тщательная настройка (привязка потоков, подбор grain size, выбор schedule) могла бы улучшить их показатели.

### Что можно улучшить

**Общие для всех версий:** применить SIMD-векторизацию для ускорения на уровне инструкций, использовать кэш-блокировку для улучшения локальности данных, попробовать объединить два прохода по памяти в один.

**Для OpenMP:** явно задать `schedule` с оптимальным размером чанка, настроить привязку потоков через `OMP_PROC_BIND`, использовать редукцию с пользовательским типом.

**Для TBB:** подобрать оптимальный `grainsize`, ограничить число потоков через `tbb::global_control`, попробовать конвейерную обработку через `parallel_pipeline`.

**Для STL:** создать пул потоков для устранения затрат на создание/уничтожение, добавить динамическую балансировку нагрузки.

**Для гибридной версии:** заменить два вызова `MPI_Allreduce` на один с пользовательской структурой, использовать асинхронные MPI-вызовы для наложения коммуникаций и вычислений, рассмотреть односторонние операции RMA.
