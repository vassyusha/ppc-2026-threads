## 1. Контекст

Из последовательной версии (SEQ) заимствуется вычислительная логика: поиск минимального и максимального значений
пикселей с последующим линейным преобразованием каждого элемента в диапазон [0, 255].

Реализация на базе стандартной библиотеки потоков (`std::thread`) позволяет исследовать производительность
многопоточности без использования высокоуровневых абстракций, таких как планировщики TBB или директивы OpenMP. Это даёт
возможность оценить реальные накладные расходы на создание и завершение потоков, а также проанализировать эффективность
прямого управления ресурсами процессора.

В отличие от OpenMP и TBB, где разбиение данных и синхронизация автоматизированы, `std::thread` предоставляет полный
контроль над разбиением диапазонов, синхронизацией и управлением жизненным циклом потоков.

## 2. Постановка задачи

Задача — повысить контрастность входного изображения за счёт выполнения линейного растяжения гистограммы. В качестве
базовой линии (baseline) используется результат последовательной версии из `seq/report.md` (время выполнения: 0.81182274
с).

Детали постановки задачи можно найти в `seq/report.md` в разделе `2. Постановка задачи`.

## 3. Базовый алгоритм

Последовательная версия за один проход находит минимальное и максимальное значения пикселей во входном массиве, после
чего за второй проход преобразует каждый пиксель по формуле линейного растяжения:
$out[i] = \frac{(in[i] - min_v)*255}{max_v-min_v}$. Если `min == max` (монохромное изображение), то возвращается копия
входного массива без преобразования.

## 4. Схема распараллеливания

- **Деление данных:** весь массив делится на $P$ равных сегментов (где $P$ — число потоков). Каждый поток получает свой
  диапазон индексов `[begin, end)` с помощью функции `GetThreadRange`.

- **Действия потоков:** в рамках задачи выполняются две основные фазы: **Поиск min/max:** каждый поток заполняет свои
  локальные переменные `current_min` и `current_max` для закреплённого за ним диапазона. **Преобразование:** после
  вычисления глобальных min/max (выполняется последовательно после объединения локальных результатов) потоки параллельно
  преобразуют элементы своего диапазона и записывают результат в выходной массив.

- **Локальные результаты:** хранятся в векторах `local_min` и `local_max` размера `num_threads`, где каждая ячейка
  принадлежит строго одному потоку. Это исключает гонки при записи.

- **Join:** выполняется дважды: после завершения первой фазы и после второй фазы. Это гарантирует, что все потоки
  завершили свою часть работы перед переходом к следующей стадии.

- **Синхронизация:** используется явная синхронизация через барьер `join()`. Явные мьютексы или атомарные операции не
  требуются, так как потоки пишут в независимые области памяти (разные индексы векторов `local_min/local_max` и разные
  диапазоны выходного массива).

## 5. Детали реализации

- **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

- **Карта поток → диапазон:** Для обеспечения равномерной загрузки используется статическая декомпозиция данных. Входной
  вектор размером $N$ делится на $P$ частей, где $P$ — количество потоков.

    **Расчёт параметров:**
  - `chunk = size / num_th` — базовый размер чанка.
  - Для потока $i$ (`i ∈ [0, P-1]`):
  - `begin = i * chunk`
  - `end = (i == P-1) ? size : begin + chunk`

                **Обработка остатка:** Последний поток ($i = P-1$) всегда получает верхнюю границу `size`, что гарантирует корректную обработку «хвоста» данных, если $N$ не кратен $P$.

- **Локальные аккумуляторы:** для сбора локальных результатов используются векторы `local_min` и `local_max` размера
  `num_threads`, инициализированные значениями `255` и `0` соответственно. Каждый поток записывает найденные минимум и
  максимум в свою уникальную ячейку по индексу `thid`, что исключает гонки благодаря непересекающимся областям памяти.

- **Схема объединения:** после завершения всех потоков (явный вызов `join()` для каждого) родительский поток
  последовательно объединяет локальные результаты в цикле, используя `std::min` и `std::max`. Объединение выполняется
  после барьера `join`, поэтому синхронизация во время работы потоков не требуется.

- **Работа с памятью:** входной массив доступен всем потокам только на чтение (безопасно). Выходной массив заполняется
  во второй фазе: каждый поток записывает данные в свой уникальный диапазон. Локальные векторы `local_min` и `local_max`
  аллоцируются один раз на этапе `RunImpl`, дополнительных выделений памяти не производится.

## 6. Проверка корректности

Результат работы STL-версии считается корректным при выполнении одного из двух условий:

- Выходное изображение монохромное (`min == max`) — тогда все пиксели остаются неизменными.
- Выходное изображение имеет полный динамический диапазон: минимальное значение равно 0, максимальное значение
  равно 255.

Помимо этого производилось сравнение с работой SEQ версии. Ошибок в логике работы алгоритма выявлено не было.

Набор функциональных тестов:

1. **Изображение 1×1 (один пиксель):** проверка граничного случая минимального размера и ветки `min_v == max_v`.

2. **Изображение 5×5:** проверка правильности растяжения на малом размере.

3. **Изображение 13×13:** проверка работы алгоритма на среднем размере входных данных.

4. **Изображение 70×70:** проверка устойчивости при большем количестве пикселей.

5. **Изображение 200×200:** проверка корректности на значительном объёме данных.

6. **Реальное монохромное изображение:** — проверка ветки `min_v == max_v` на реальных данных, загружаемых из файла.

### Анализ результатов и масштабируемости

1. **Эффективность параллелизации:** Применение `std::thread` позволило сократить время обработки изображения с **0.81
   с** до **0.17 с**. Пиковое ускорение **4.76×** достигнуто на 8 потоках.

2. **Масштабируемость и ограничения:** До 4 потоков наблюдается уверенный рост производительности (ускорение достигло
   3.02×). Дальнейшее увеличение числа потоков до 8 даёт прирост лишь до 4.76×. Такая динамика характерна для **Memory
   Bound** задач: пропускная способность памяти становится узким местом при одновременной работе нескольких ядер.

3. **Анализ эффективности (Efficiency):**

- На 2 потоках эффективность составляет **0.94**, что говорит о минимальных издержках на статическое разбиение данных.

- Снижение эффективности до **0.59** на 8 потоках связано с гетерогенной архитектурой процессора (P- и E-ядра). Разница
  в производительности ядер приводит к неравномерной загрузке и простою более быстрых потоков перед барьером `join`.

4. **Влияние накладных расходов:** Затраты на создание, запуск и объединение потоков становятся всё более ощутимыми с
   ростом их числа, что вносит дополнительный вклад в снижение эффективности.

**Вывод:** STL Threads обеспечивает хорошее ускорение задачи. Рекомендуемое число потоков — 4, дающее ускорение
**3.02×** при эффективности **76%**.

## 9. Выводы

STL-версия на `std::thread` продемонстрировала устойчивое ускорение до **4.76×** на 8 потоках, сократив время обработки
изображения с 0.81 с до 0.17 с.

Ручное управление потоками позволяет получить полный контроль над разбиением данных и синхронизацией, избегая накладных
расходов на скрытые проверки и очереди задач, присущие высокоуровневым библиотекам. Однако такая реализация требует
более тщательного подхода к написанию кода для предотвращения ошибок доступа к памяти и состояния гонки.

Основным ограничением масштабируемости является пропускная способность памяти (Memory Bound). Оптимальным для данной
задачи является использование **4 потоков**, обеспечивающих ускорение **3.02×** при эффективности **76%**.
