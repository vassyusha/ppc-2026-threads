## 1. Контекст

Из последовательной версии (SEQ) переносится вычислительное ядро: нахождение минимального и максимального значений пикселей с последующим линейным преобразованием каждого элемента в диапазон [0, 255].

Для реализации параллельной версии выбрана библиотека oneTBB. Данный выбор обусловлен потребностью в гибкой системе планирования задач, которая самостоятельно распределяет нагрузку между потоками. В отличие от OpenMP, TBB предоставляет более тонкий контроль над разбиением данных на блоки и их распределением, что особенно важно для задач, ограниченных пропускной способностью памяти. Кроме того, встроенные алгоритмические шаблоны `parallel_reduce` и `parallel_for` точно соответствуют структуре алгоритма — редукционной операции поиска min/max и независимому преобразованию каждого пикселя.

## 2. Постановка задачи
Задача — повысить контрастность входного изображения за счёт выполнения линейного растяжения гистограммы. 
В качестве базовой линии (baseline) используется результат последовательной версии из `seq/report.md` (время выполнения:  0.81182274 с).

Детали постановки задачи можно найти в `seq/report.md` в разделе `2. Постановка задачи`.

## 3. Базовый алгоритм
Последовательная версия за один проход находит минимальное и максимальное значения пикселей во входном массиве, после чего за второй проход преобразует каждый пиксель по формуле линейного растяжения: $out[i] = \frac{(in[i] - min_v)*255}{max_v-min_v}$. Если `min == max` (монохромное изображение), то возвращается копия входного массива без преобразования.

## 4. Схема распараллеливания

- **Выбранные примитивы:**
  - Для поиска min/max применяется `tbb::parallel_reduce`, как редукционная операция.
  - Для преобразования пикселей или копирования в монохромном случае используется `tbb::parallel_for`. 

- **Диапазон работы:**
  - Используется `tbb::blocked_range<size_t>(0, size)`, где `size` — общее количество пикселей.
  - Диапазон автоматически разбивается на поддиапазоны, каждый из которых обрабатывается отдельным потоком.

- **Grainsize:**
  - Используется значение `grainsize` по умолчанию. Библиотека TBB достаточно оптимизирована, чтобы автоматически подбирать оптимальный размер зерна, балансируя между накладными расходами на планирование и эффективностью использования кэша.

- **Partitioner:** используется `auto_partitioner` по умолчанию, распределяющий блоки по доступным ядрам.



## 5. Детали реализации
- **Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

- **Локальные аккумуляторы:** для каждого потока в `parallel_reduce` создаётся локальная копия структуры `MM`, содержащей минимум и максимум. Это предотвращает конфликты доступа и исключает необходимость синхронизации при обновлении значений. Каждый поток независимо обрабатывает свой диапазон, сохраняя локальные результаты в собственной копии структуры. Использование структуры `MM` позволяет использовать один проход `parallel_reduce` вместо двух для поиска значений `min_v` и `max_v`.

- **Объединение результатов:** после завершения обработки своих диапазонов TBB автоматически вызывает редукционную функцию, которая попарно объединяет локальные структуры `MM`, вычисляя глобальные минимум и максимум. Это исключает состояние гонки и делает код безопасным без использования атомарных операций или критических секций.

- **Работа с памятью:** алгоритм использует минимальное количество дополнительной памяти — только входной и выходной массивы. В отличие от STL-версии, не требуется выделять векторы локальных результатов (`local_min`, `local_max`) и управлять потоками вручную. TBB самостоятельно управляет временными структурами внутри примитивов `parallel_reduce` и `parallel_for`, что упрощает код и снижает накладные расходы на выделение памяти. Дополнительные аллокации отсутствуют, так как все необходимые буферы создаются и уничтожаются внутри TBB-примитивов без участия пользователя.

- **Риски гонок:** устранены за счёт того, что `parallel_reduce` работает с локальными копиями аккумуляторов у каждого потока, а `parallel_for` обеспечивает непересекающиеся диапазоны записи в выходной массив. Благодаря этому гонки при обновлении данных полностью исключены.

## 6. Проверка корректности
Результат работы TBB-версии считается корректным при выполнении одного из двух условий:
- Выходное изображение монохромное (`min == max`) — тогда все пиксели остаются
  неизменными.
- Выходное изображение имеет полный динамический диапазон: минимальное значение
  равно 0, максимальное значение равно 255.

Помимо этого производилось сравнение с работой SEQ версии. Ошибок в логике работы алгоритма выявлено не было.


Набор функциональных тестов:
1. **Изображение 1×1 (один пиксель):** проверка граничного случая
   минимального размера и ветки `min_v == max_v`.

2. **Изображение 5×5:** проверка
   правильности растяжения на малом размере.

3. **Изображение 13×13:** проверка работы алгоритма на среднем размере входных данных.

4. **Изображение 70×70:** проверка устойчивости при большем количестве
   пикселей.

5. **Изображение 200×200:** проверка корректности
   на значительном объёме данных.

6. **Реальное монохромное изображение:** — проверка ветки `min_v == max_v` на реальных данных,
   загружаемых из файла.

## 8. Результаты
Эксперименты проводились на векторе **400 000 000** элементов типа `uint_8t`.

| Threads | Время (s) | Ускорение ($S$) | Эффективность ($E$) |
| :-----: | :--------: | :-------------: | :-----------------: |
| 1 (SEQ) |    0.81182274   |      1.00       |        1.00         | 
|    2    |    0.42426248   |      1.91       |        0.95         | 
|    4    |    0.23717768   |      3.42       |        0.85         | 
|    8    |    0.14896833   |      5.45       |        0.68         | 

_Где:_

- $S = T_{seq} / T_th$ — коэффициент ускорения.
- $E = S / threads$ — эффективность использования вычислительных ресурсов.

1. **Ускорение:** Производительность уверенно растёт с увеличением числа потоков вплоть до 8.
   На 4 потоках достигнуто ускорение $3.42\times$, на 8 потоках — $5.45\times$, что является высоким
   результатом для алгоритмов с интенсивными операциями ввода-вывода в память.

2. **Эффективность:** При использовании 2 потоков эффективность составляет $0.95$,
   что говорит о минимальных издержках на организацию параллельных вычислений. На 4 потоках
   эффективность держится на уровне $0.85$, а на 8 потоках снижается до $0.68$, что ожидаемо
   при возрастающей конкуренции за ресурсы кэш-памяти.

3. **Точка насыщения:** Переход с 4 на 8 потоков даёт прирост ускорения почти в 1.6 раза
   ($3.42 \rightarrow 5.45$). Дальнейшее увеличение числа потоков выше 8, вероятно, приведёт
   к более заметному падению эффективности из-за ограничений пропускной способности памяти.

4. **Сравнение с SEQ:** Применение TBB сократило время обработки
   изображения с $0.81$ секунды до $0.15$ секунды, что соответствует ускорению более чем в 5 раз.

**Итог:** TBB обеспечивает высокую масштабируемость на данной задаче. Оптимальной конфигурацией
является использование 4–8 потоков, где достигается хороший компромисс между скоростью обработки
и эффективностью утилизации вычислительных ресурсов.

## 9. Выводы
TBB-версия обеспечивает устойчивое ускорение (до $5.45\times$ на 8 потоках) благодаря эффективному использованию `parallel_reduce` для поиска min/max и `parallel_for` для преобразования пикселей.

**TBB удобнее** при работе с редукционными операциями, неравномерной нагрузке (автоматическая балансировка задач) и вложенном параллелизме, где требуется гибкое управление задачами.

**Overhead TBB становится заметным** при обработке малых объёмов данных (менее 10–50 тыс. элементов), когда затраты на создание и распределение задач превышают выигрыш от параллелизации, а также в сильно связанных алгоритмах с частой синхронизацией.

Основное ограничение масштабируемости для данной задачи — пропускная способность памяти. Оптимальное число потоков составляет 4–8 в зависимости от архитектуры целевой системы.
